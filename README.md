# Scripts Repository

Репозиторий со скриптами для работы с аналитикой и обработки данных.

## 📁 Структура

- `egrul_report.py` - Генерация отчетов по ЕГРЮЛ из ClickHouse PROD
- `llm_tag_experiment/` - Эксперименты с LLM-классификацией банковских транзакций
- `llm_statement_preprocessor/` - Парсер банковских выписок (RTF/Excel/CSV) с использованием LLM

## 🚀 Быстрый старт

### 1. Клонирование репозитория

```bash
git clone https://github.com/uguwap/scripts.git
cd scripts
```

### 2. Установка зависимостей

```bash
# Общие зависимости
pip install -r requirements.txt

# Для llm_statement_preprocessor
cd llm_statement_preprocessor
pip install -r requirements.txt
```

### 3. Настройка переменных окружения

1. Скопируйте `.env.example` в `.env`:
   ```bash
   cp .env.example .env
   ```

2. Откройте `.env` и заполните все необходимые переменные (см. `VARIABLES.md`)

**⚠️ ВАЖНО:** Файл `.env` уже добавлен в `.gitignore` и не будет загружен в репозиторий. Никогда не коммитьте файл `.env` с реальными паролями!

## 📋 Подробное описание скриптов

### 1. egrul_report.py

**Назначение:** Генерация Excel-отчетов по данным ЕГРЮЛ из ClickHouse PROD для анализа качества запросов к API и статистики по контрагентам.

**Как работает:**

1. **Подключение к БД:**
   - Устанавливает SSH туннель через `doc.ai-referent.ru` к внутреннему ClickHouse серверу
   - Использует `paramiko` для создания SSH соединения
   - Пробрасывает локальный порт `19000` на удаленный `10.10.0.4:9000`
   - Подключается к ClickHouse через туннель

2. **Сбор данных:**
   - Выполняет 11 SQL-запросов к ClickHouse (только SELECT, без изменений)
   - Собирает статистику по:
     - Сводка: общее количество записей в `counterparties_info`, `person_info`, `relations`, `relations_edges`
     - По дням: динамика загрузки данных за последние 30 дней
     - По проектам: покрытие контрагентов по проектам
     - Цепочки по проектам: анализ возможных пар для построения цепочек
     - Статусы: распределение статусов контрагентов
     - Типы связей: статистика по типам связей в `relations`
     - Аудит качества: валидация ИНН, дубли, мусорные данные
     - Качество запросов API: оценка запросов к разным ручкам API ЕГРЮЛ
     - Дубли запросов: ИНН, которые запрашивались несколько раз
     - Мусорные запросы: невалидные ИНН, на которые были сделаны запросы

3. **Формирование отчета:**
   - Создает Excel файл с несколькими листами
   - Применяет форматирование (заголовки, границы, автоширина колонок)
   - Добавляет итоговые строки и метаданные (дата генерации, период)

**Использование:**
```bash
python egrul_report.py
```

**Выходной файл:** `egrul_report_YYYY-MM-DD.xlsx`

**Требуемые переменные окружения:**
- `SSH_HOST`, `SSH_PORT`, `SSH_USER`, `SSH_PASSWORD` - для SSH туннеля
- `CH_REMOTE_HOST`, `CH_REMOTE_PORT`, `CH_USER`, `CH_PASSWORD`, `CH_DATABASE` - ClickHouse PROD
- `DAYS_BACK` - количество дней для анализа (по умолчанию 30)

**Почему SSH туннель:**
ClickHouse PROD находится во внутренней сети и недоступен напрямую. SSH туннель обеспечивает безопасное подключение через промежуточный сервер.

### 2. llm_tag_experiment/run_classification.py

**Назначение:** Классификация банковских транзакций по трёхуровневой иерархии тегов с использованием LLM (DeepSeek).

**Как работает:**

1. **Подключение к БД:**
   - Прямое HTTP подключение к ClickHouse DEV (`dev.ai-referent.ru:8123`)
   - Использует HTTP интерфейс ClickHouse (не требует SSH туннеля)

2. **Выгрузка данных:**
   - Загружает транзакции из `ai_referent_merge` для проекта "БОРХИММАШ ФУЛЛ ПРОЕКТ"
   - Обогащает данные ОКВЭД контрагентов из `counterparties_info`
   - Определяет лизинговые компании по ОКВЭД кодам (64.91, 77.11, 77.31, 77.33, 77.39)
   - Может работать с выборкой (`SAMPLE_SIZE`) или всеми данными

3. **LLM-классификация:**
   - Разбивает транзакции на батчи (по умолчанию 40 транзакций)
   - Для каждого батча формирует промпт с описанием транзакций
   - Отправляет запрос к LLM API (DeepSeek через `https://neuro.sspb.ru/v1`)
   - Парсит JSON-ответ с тегами для каждой транзакции
   - Обрабатывает ошибки и повторяет запросы при необходимости

4. **Классификатор:**
   - **Уровень 1:** "Поступления" или "Списания"
   - **Уровень 2:** Категория (например, "От заказчиков/покупателей", "Платежи контрагентам")
   - **Уровень 3:** Подкатегория (например, "Транспортные услуги", "Топливо, ГСМ")

5. **Сохранение результатов:**
   - CSV файл с транзакциями и тегами
   - Текстовая сводка со статистикой по тегам

**Использование:**
```bash
cd llm_tag_experiment
python run_classification.py
```

**Выходные файлы:**
- `output/classification_YYYYMMDD_HHMMSS.csv` - детальные результаты
- `output/summary_YYYYMMDD_HHMMSS.txt` - сводная статистика

**Требуемые переменные окружения:**
- `CH_DEV_HOST`, `CH_DEV_PORT`, `CH_DEV_USER`, `CH_DEV_PASSWORD`, `CH_DEV_DATABASE` - ClickHouse DEV
- `LLM_API_KEY`, `LLM_BASE_URL`, `LLM_MODEL` - настройки LLM API
- `BATCH_SIZE` - размер батча (по умолчанию 40)
- `LLM_TIMEOUT` - таймаут запросов (по умолчанию 180 сек)
- `LLM_MAX_RETRIES` - максимум повторов (по умолчанию 3)
- `SAMPLE_SIZE` - размер выборки (0 = все данные, >0 = случайная выборка)

**Почему DEV база:**
Эксперименты проводятся на DEV окружении, чтобы не нагружать PROD и иметь возможность тестировать на реальных данных без риска.

### 3. llm_statement_preprocessor/

**Назначение:** Парсер банковских выписок (RTF, Excel, CSV) с использованием LLM для извлечения структурированных данных в стандартный Excel формат для дальнейшей обработки нормалайзером.

**Архитектура:**

1. **Входные данные:**
   - RTF файлы (конвертируются в текст через `striprtf`)
   - Excel файлы (читаются через `openpyxl`)
   - CSV файлы

2. **Обработка:**
   - Разделение на шапку (header) и табличную часть (operations)
   - Разбиение операций на батчи (по умолчанию 10 строк)
   - Параллельная обработка: LLM + Recovery парсер

3. **LLM обработка:**
   - Отправка батчей в LLM для структурирования
   - Поддержка нескольких провайдеров: Claude, OpenAI, DeepSeek, Groq, Gemini
   - Извлечение структурированных данных (дата, сумма, назначение, ИНН и т.д.)

4. **Recovery парсер:**
   - Детерминированный парсер табличного текста
   - Работает параллельно с LLM как страховка
   - Используется при ошибках LLM, пустых ответах, невалидном JSON
   - Гарантирует сохранение всех операций

5. **Проверка целостности:**
   - Сверка количества операций с ожидаемым
   - Проверка наличия `source_line` (номер строки из исходного файла)
   - Автоматический добор пропущенных операций из recovery
   - В `strict_mode=true` переключение на recovery при нерешаемых проблемах

6. **Выходные данные:**
   - Стандартный Excel файл с листами:
     - **Header** - шапка выписки
     - **Operations** - операции
     - **Debug** - отладочная информация с `source_line`

**Использование:**

**Через API (рекомендуется):**
```bash
cd llm_statement_preprocessor
python api.py
```

API доступен на `http://localhost:9000/docs` (Swagger UI)

**Через CLI:**
```bash
python main.py <путь_к_файлу>
```

**Требуемые переменные окружения:**
- `LLM_PROVIDER` - провайдер LLM (claude, openai, gemini, deepseek, groq)
- `LLM_API_KEY` - API ключ
- `LLM_MODEL` - модель (по умолчанию зависит от провайдера)
- `LLM_MAX_TOKENS` - максимум токенов (по умолчанию 4096)
- `LLM_TEMPERATURE` - температура (0.0 для детерминированного вывода)
- `LLM_ENABLED` - включить/выключить LLM (1/0)
- `BATCH_SIZE` - размер батча (по умолчанию 10)

**Режим без LLM:**
Если `LLM_ENABLED=0` или нет API ключа, работает только recovery парсер. Это медленнее, но гарантирует извлечение всех операций.

**Почему два парсера:**
- LLM лучше справляется с нестандартными форматами и извлекает больше информации
- Recovery парсер гарантирует, что ничего не потеряется при ошибках LLM
- Вместе они обеспечивают максимальное качество и надежность

**Логи и отладка:**
Все артефакты сохраняются в `logs/<run_id>/`:
- Исходные тексты (header, operations)
- Промпты и ответы LLM для каждого батча
- Результаты recovery парсера
- Причины выбора LLM или recovery для каждого батча
