### LLM Statement Preprocessor (RTF/XLSX → стандартный XLSX для нормалайзера)

Сервис читает **RTF/Excel/CSV** “неудобные” выписки, извлекает шапку/операции (через LLM) и сохраняет итог в Excel‑формате, который понимает `bank-statement-normalizer`.

Ключевые свойства:
- **Ничего не теряем**: если LLM падает на батче или даёт рассинхрон — батч автоматически добирается/подменяется recovery‑парсером.
- **Сверка с первоисточником**: у каждой операции есть `source_line` (номер строки `Строка N:` из входного текста). В итоговом файле это видно на листе **Debug**.
- **Checkpoints**: можно сохранять промежуточные Excel‑файлы каждые N батчей.

---

### Что такое recovery (и почему это не противоречит “LLM по задумке”)

**Recovery** — это детерминированный парсер табличного текста, который извлекает операции без LLM. Он нужен не как “вторая система”, а как **страховка качества**:
- LLM может вернуть пустой ответ, невалидный JSON или пропустить часть строк (особенно на длинных файлах).
- Провайдер может вернуть 402/квоту/временную ошибку.
- В таких ситуациях остановка пайплайна недопустима: лучше “дольше, но правильно”.

Текущая стратегия:
- **Пробуем LLM** как основной источник структуры.
- **Параллельно строим recovery** на том же батче.
- Считаем, сколько операций ожидается по источнику (`expected_ops`) и проверяем наличие `source_line`.
- Если LLM “не сходится” (дырки по `source_line` / низкое покрытие) — добираем пропуски из recovery или целиком выбираем recovery в `strict_mode=true`.

---

### Что было не так (предыдущий неудачный кейс)

1) Потеря первой операции
- Детект начала операций работал только для строк, где дата в начале (`dd.mm.yy`), а в табличном RTF дата стоит во второй колонке (`│ N │ dd.mm.yy │ ...`).
- Из‑за этого шапка/операции могли разделиться неправильно и часть операций уходила в “шапку”.

2) “Разъезд” данных и потеря операций на финальных батчах
- В recovery‑парсере пустые ячейки выкидывались при разборе строки таблицы, из‑за чего **съезжали позиции колонок** и поля начинали попадать “не туда”.
- При ошибках провайдера (например, 402) без корректного фоллбэка пайплайн мог резко деградировать по качеству.

3) Невозможность честной сверки с первоисточником
- `source_line` не заполнялся на recovery‑пути, а LLM мог его не вернуть.

---

### Что исправлено (наши правки)

1) Корректный детект начала операции для табличных RTF
- `looks_like_operation_start` умеет распознавать строки вида `│ N │ dd.mm.yy │ ...`.

2) Исправлен recovery‑парсер (табличные колонки больше не “съезжают”)
- Пустые ячейки внутри строки сохраняются, убираются только граничные разделители.

3) `source_line` гарантированно сохраняется
- Recovery проставляет `source_line` из `Строка N:`.
- Для LLM‑операций `source_line` пытаемся восстановить по совпадению с recovery.

4) “Железобетонная” целостность батча
- Для каждого батча собираем ожидаемые `source_line`, добираем пропуски из recovery.
- В `strict_mode=true` при нерешаемых дырах батч переключается на recovery целиком.
- Решение по батчу пишется в `selection.json`.

### Быстрый запуск (Windows PowerShell)

Перейти в директорию сервиса:

```powershell
cd C:\project\scripts\llm_statement_preprocessor
```

Активировать виртуальное окружение (если используете репозиторное `venv/`):

```powershell
.\venv\Scripts\Activate.ps1
```

Установить зависимости:

```powershell
python -m pip install -r .\requirements.txt
```

Задать переменные окружения (пример для DeepSeek):

```powershell
$env:LLM_PROVIDER="deepseek"
$env:LLM_API_KEY="sk-..."
# опционально:
$env:LLM_MODEL="deepseek-chat"
$env:LLM_MAX_TOKENS="4096"
$env:LLM_TEMPERATURE="0"
$env:BATCH_SIZE="10"
```

Запустить API:

```powershell
python .\api.py
```

Открыть Swagger UI:
- `http://localhost:9000/docs`

Проверка состояния:
- `http://localhost:9000/health`

---

### Запуск без LLM (только recovery‑режим)

Если ключа нет или LLM недоступен — можно запустить recovery‑режим:

```powershell
$env:LLM_ENABLED="0"
python .\api.py
```

В этом режиме:
- шапка будет пустой/частично заполненной (зависит от входа),
- операции извлекаются из табличного текста recovery‑парсером,
- `source_line` сохраняется.

---

### Как пользоваться API

Метод:
- `POST /preprocess/statement` (multipart/form-data)

Поля формы:
- `file`: файл `.rtf` / `.xlsx` / `.xls` / `.csv`
- `enable_checkpoints` (default `true`): сохранять ли checkpoints
- `checkpoint_interval` (default `50`): каждые сколько батчей сохранять файл
- `strict_mode` (default `true`): режим “строже” (используется для логов/подозрительных батчей)

Ответ:
- `final`: ссылка на итоговый `final_all_operations.xlsx`
- `checkpoints`: список checkpoint‑файлов
- `logs_dir`: директория с логами прогона (input/attempts/recovery)
- `recovery_batches`: сводка по батчам, где сработал recovery

---

### Где смотреть, если “что-то поехало”

Артефакты прогона пишутся в `scripts/llm_statement_preprocessor/logs/<run_id>/`:
- `source_header.txt`, `source_operations.txt` — исходный текст после конвертации
- `batches/batch_XXXX/input.txt` — текст батча
- `batches/batch_XXXX/attempt_YY/` — промпт/ответ LLM и ошибки JSON‑парсинга
- `batches/batch_XXXX/recovery.json` — результат recovery (если LLM упал/вернул пусто)
- `batches/batch_XXXX/selection.json` — почему выбрали `llm` или `recovery` (покрытие, expected_ops, source_line)
- `summary.json` — итог (в т.ч. причина отключения LLM при ошибках баланса/квоты)

Итоговый Excel:
- `uploads/final_all_operations.xlsx` (лист **Debug** показывает `source_line` → строку Excel)
